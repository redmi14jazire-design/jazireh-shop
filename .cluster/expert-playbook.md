# Agent 集群模式 Playbook(进 S2 / S5 前读)

## 工作台文件约定(.cluster/<taskId>/,taskId = 本次任务 id)
- plan.md:派单清单。每条子任务 → 负责 worker → 预期产物文件名 → 依赖;外加复核项、交付项。
- worker_NN.md:worker 回传。格式:结论 / 证据 / 分析 / 缺口与风险 / 建议入档位置。
- review.md:复核结论。四级置信:High(≥2 独立源一致) / Medium(1 权威源) / Low(弱源) / Conflict(冲突,显式列出不抹平)。
- brief.md:交付前的文档 brief。
- DELIVERY/:最终成品文件(系统会查这个目录确认真交付,务必把成品写进来)。

## S2 派单细则
- 一条子任务一个 worker。例外只有一种:一次性取数/跑命令、拆出去只多一趟来回的,留主线自己做。
- 长文写作:writer 章节数 ≥ 调研维度数。"调研派 N 路、写作只派 1 路"是被禁止的结构——规划时就逐章拆出 writer。
- 画依赖图:被依赖的子任务排前轮;摘要/结论/综合类排在它们总结内容之后的轮次。

## S3 worker prompt 模板
开头一句固定:"你不是一个人在干,别动不属于你的产物。"
给全三样:
1. 任务边界:做什么、不做什么、产物 write 到哪个 worker_NN.md。
2. 它拿不到、你给它的上文:主线已确认的事实与前序结论;写作 worker 另给与本章相关的素材文件路径 + 用户原始源路径(按章节选发,不全量灌——提取会丢精确数字/公式/表格,原始源要到达 writer)。
3. 回传格式:结论 / 证据 / 分析 / 缺口与风险 / 建议放入最终 DOCX 的位置。没证据就写没有,不把猜测伪装成结论。
联网 worker:走 autoglm-websearch/open-link,不要使用 OpenClaw 内置 web_search/web_fetch;只引实际检索到的来源,查不到写明缺口、不凭记忆补。
worker 用 skill 时 task 里引 skill 名让它自己加载,别 dump 内容。worker 默认不生成最终 Office 文件(被明确派去产图表/数据表等中间子产物的除外)。

## S4 复核细则
判断要不要起复核员:产出含综合得出的结论/预测/假设/价值判断、或非琐碎新代码/不可逆操作、或用户会据此行动——命中任一就起。
分维度、单维审(多次单维优于一次模糊"看一遍"):事实 / 逻辑 / 数字 / 代码 各算一块,互相独立的并行各派一个复核员。
复核员也是 spawn 的 worker,结论 write 进 review.md。

## S5 交付细则
- 形态:用户指定就照办;没指定,一两段讲得完 → 纯文本回复,否则默认 `docx`。
- GLM-Office 路由:报告/方案/纪要/白皮书/调研 → `docx`;演示/deck/路演 → `ppt`;表格/数据模型/Excel → `xlsx`;固定版式/打印/合同/论文终稿 → `pdf`;图表/流程图/架构图 → `charts` 再嵌入。
- `delivery-artifact` 只处理 HTML / 网页 / 交互原型 / Stage / landing page;别因为成果长就走它。
- 调 GLM-Office skill 前先确认它在 `<available_skills>` 中存在;不可用就回退到 `write-skill` 产 markdown 并说明降级原因。
- 长文:先用 `write-skill` 多 worker 分章产干净 markdown(每章一 writer + 审稿闸门),章节落工作台,最终文件再由 `docx` 生成。deep-research 跑完默认先过 write-skill 分章成稿再走 DOCX,不是你自己综合(例外:用户明确"只要素材"或指定别的格式)。
- 执行摘要在所有章节定稿后最后写(它总结实际产出,不是计划)。
- 组装严格 UTF-8,不加 errors 容错;解码失败重派该章 writer,绝不用 gbk/latin-1/errors=ignore(会变成静默乱码)。
- 引用呈现跟载体:正文事实用行内可点 `[出处](url)`;长文档(DOCX/PDF)用脚注角标 `[^n^]` + 文末来源表。
- brief.md 含:标题、受众、目的、执行摘要、章节大纲、带来源的关键发现、表格/图表规格、风险假设、附录候选、语气格式。不要把 worker 产物原样拼接进 DOCX。
- 质量门:有标题/摘要/章节结构;有结论和 so-what;关键事实有来源;表格/列表/图表组织好;风险/假设/未覆盖范围单列;按 `docx` skill 工序做 postcheck。数据/财务类数字口径(单季/累计/TTM)要对。
- 成品文件写进 DELIVERY/,收尾发一次最终进度简报标全部完成,回复正文最后一行写终态标记。
- 交付物清单形式:每个成品用可点 markdown 链接 `[用途名](绝对路径)` 列出(路径必须是 DELIVERY/ 下的完整绝对路径,不是相对路径、不是省略号截断名)。禁止把成品文件名放进表格单元格或 `代码块` 当链接用——那样前端识别不出文件、用户点不开也在文件区找不到。表格只用来对比属性,不用来当文件入口。
- 收尾纪律(最后一条消息一次做完):① 重发完整 `<plan>`,所有步骤(含审稿/拼装/生成文件)都标 `done`——某步降级了也标 done 并在交付里说明;② 给交付物清单(按上一条用 markdown 链接);③ 末行 [[final:submit_result]]。绝不输出"需要我做什么后续处理 / 要不要导出PDF / 转飞书 / 还要补充吗"这类反问后停手——导出/转换/补充都属于本轮交付,直接做或直接收尾,不反问。
